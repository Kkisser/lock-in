## 5. Behavior of the “Now” section

The “Now” section should work like this:

### If there is no active session
Open the **event tree**.

### If there is already an active session
When the user enters “⚡ Now”, it should **immediately open the active session screen**.

This is a fixed requirement.

---

## 6. Event model

There is an event tree.

It can contain:

- **groups**
- **subgroups**
- **actions**

### Rules:
- a group can contain:
  - subgroups
  - actions
- an action can exist:
  - inside a group
  - without a group, at the root level
- only an **action** can be started
- an action can have **many sessions**

### Example:
- Work
  - Client A
    - Backend
    - Frontend
  - Project B
- Study
  - Python
  - SQL
- Walking

---

## 7. Event tree — important UI implementation detail

The event tree must be implemented as **inline navigation inside a single message**.

That means:

- the bot shows one message
- this message contains the list of elements on the current level
- when the user enters a group, the bot **edits this same message**
- the content is replaced with the content of the selected group
- when the user presses “Back”, the bot returns to the previous tree level
- do not create a new message for every tree level

### The tree screen must show:
- current path
- list of groups and actions
- `🔙 Back` button (if not at root)
- `🏠 Main menu` button

### Element types:
- `📁` — group
- `▶️` — action

### For groups, show:
- the number of **direct child elements**
  - subgroups
  - actions

### Actions without a group
Show them directly at the current level next to groups, **without a separate virtual “Ungrouped” folder**.

Example:
- 📁 Work (3)
- 📁 Study (2)
- ▶️ Walking
- ▶️ Reading

## 8. Interface states inside the “Now” section

There are 3 interface states inside “Now”:

### 1. Event tree navigation
When there is no active session.

### 2. Selected action
When the user reaches a specific action.

The selected action screen should show:
- action name
- path / parent
- how much time was already spent on this action **today**

Buttons:
- `▶️ Start`
- `🔙 Back`
- `🏠 Main menu`

### 3. Active session
When the action has already been started and the timer is running.

## 9. Session start logic

Only an **action** can be started.

When the user presses `Start`:

### If there is no active session
Create a new session.

### If there is already an active session
Show this message:

> You already have an active action  
> State: Now

And then redirect the user to the active session screen.

---

## 10. Session limit

A user can have **only one active session at a time**.

---

## 11. Active session screen

The active session screen should display:

- action name
- state: **Now**
- status: running / paused
- elapsed time
- start time
- how much time was already spent on this action today

### When the session is running
Buttons:
- `⏸ Pause`
- `✅ Finish`
- `📊 Today`
- `🏠 Main menu`

### When the session is paused
Buttons:
- `▶️ Resume`
- `✅ Finish`
- `🏠 Main menu`

---

## 12. Live timer

The active session message should update **every 5 seconds**.

### Important:
- the final duration shown when finishing the session must be **precise**
- do not round the final result to 5-second steps

---

## 13. Very important rule

**Message update failure ≠ session loss**

If Telegram does not allow updating the message:

- the session must not be lost
- the timer must not be reset
- the database must remain the source of truth
- the live message can be recreated later if needed

---

## 14. Time storage

Do not update the database every 5 seconds.

The database should store:
- `started_at`
- session status
- pauses
- `ended_at`
- final duration after finishing (optional if you want to store it explicitly)

The current elapsed time should be calculated **on the fly**.

---

## 15. Pauses

Store pauses in a **separate table**.

One session can have multiple pauses.

---

## 16. Behavior after bot restart

If the bot is restarted:

- the active session must remain in the database
- it must not be lost
- the live message can be reopened / recreated later if necessary

---

## 17. “Today” section

The “Today” section should show:

- list of actions that had sessions today
- total time per action
- number of sessions per action
- total time for the day

Example:
- Backend API — 01:10:00 (3 sessions)
- SQL — 00:40:00 (1 session)
- Walking — 00:25:00 (2 sessions)
- Total for today: 02:15:00
